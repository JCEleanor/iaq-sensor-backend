# IAQ Sensor System Design

[Visual notes](https://excalidraw.com/#json=I8Zi_gOfSHg91hWTi9zQc,9_Ioj2GMk-AGJgyyWg0dtg)

## device_id

- **physical**: we have a MAC address, which is tied to the physical hardware
- **logical**: configurable identifier that lets us repurpose HW for other deployments

## IAQ "packet"

- `data_counter` is monotonic ACROSS BOOTS to the same `device_id`
- data counter is _contiguous_; no gaps
- `(device_id_physical, device_id_logical, data_counter, {sensor=value, ...}, device_timestamp)`
- response from server is: current server timestamp, HWM for that `device_id`

## Server High Water Mark (HWM)

- the biggest `data_counter` (for that `device_id`) that we have committed to the database
- send this back to the sensor on every insert

## Timestamps

- milliseconds since UTC Unix Epoch
- do timezone normalization on the 'frontend' when you query
- have a table that associates `device_id`s with a "deployment"
- deployment metadata includes:
  - point of contact
  - timezone
  - etc

## MVP

- postgres/timescaleDB schema
- frontend HTTP server in Go that implements the protocol
  - commit data, update HWM
  - respond with HWM, server timestamp
- Python/Go/whatever-based demo 'sensor' script that generates fake data to send to the server

## Target Load

- 10k sensors, each sensor monitoring multiple quantities
